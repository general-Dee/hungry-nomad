# Briefing: Staff/Admin App for hungry-nomad

You're building the staff-facing admin app for **hungry-nomad**, a Nigerian
food-ordering web app (Next.js + Supabase + Paystack). hungry-nomad is a
separate, existing repo that is **customer-facing only** — it has no staff
login, no order-management UI, and no status-update API. That's your job,
in this app, on your own.

You do not have access to hungry-nomad's source. This document is your
source of truth for how the two apps must interoperate. It was generated by
reading hungry-nomad's code directly (its own internal contract doc, API
routes, pricing logic, and SQL) as of 2026-07-24 — treat it as accurate for
that snapshot, but if something you observe in the live Supabase project
contradicts it, trust the live project and flag the mismatch to your user
rather than guessing.

---

### `order_items`
| column | type | notes |
|---|---|---|
| `id` | `number` | PK |
| `order_id` | `number` | FK to `orders.id` |
| `product_id` | `number` | FK to `products.id` |
| `quantity` | `number` | |

`category` is NOT persisted on `order_items`. If you need an item's category
(e.g. for reporting), join through `product_id` → `products.category`.

### RLS is default-deny for `anon` on `orders`/`order_items`

Both tables have Row Level Security enabled with **zero policies granted to
the `anon` role** — the anon key (the public, browser-shippable key) cannot
read or write these two tables via direct REST calls at all. This was a
deliberate fix (RLS was previously found disabled entirely, then briefly
too-permissive, then locked down). `products` and `delivery_zones` still
allow `anon` `SELECT` only (no writes).

**This means you cannot use hungry-nomad's public anon key to build your
admin views or status updates.** You need your own authenticated access
path — either:
- A Supabase **service-role key**, provisioned separately for your app,
  used only in server-side code (API routes/server actions), never shipped
  to the browser, or
- Supabase Auth users with an elevated role (see §5) plus RLS policies you
  write yourself that grant that role access to `orders`/`order_items`.

Either way, provision your own credentials. Do not ask for or reuse
hungry-nomad's `SUPABASE_SERVICE_ROLE_KEY`.

## 2. Order status values and legitimate transitions

The declared type is:

```ts
status: 'pending' | 'paid' | 'failed' | 'delivered'
```

What hungry-nomad's own code actually does with it:
- `'pending'` — set once, at order creation. Every order starts here.
- `'paid'` — set **only** after hungry-nomad calls Paystack's
  `/transaction/verify/:reference` endpoint server-side and confirms the
  amount Paystack returns matches the order's own server-computed
  `total_amount` (converted to kobo). If the amounts don't match, the order
  is left `pending` and nothing is written.
- `'failed'` and `'delivered'` — hungry-nomad's code never writes either of
  these. Managing orders beyond `pending`/`paid` (marking something
  `delivered`, or manually failing a stuck `pending` order) is entirely
  your job in this app.

### The hard rule: never let staff manually set `status = 'paid'`

hungry-nomad previously had a staff dashboard with exactly this
capability, and it briefly shipped a bug that allowed a manual
`pending` → `paid` transition through the staff UI — i.e. a staff member
(or anyone who compromised that login) could mark an unpaid order as paid
without any money changing hands. That whole feature was later removed
from hungry-nomad specifically so this class of bug couldn't recur there.

**Do not reintroduce this in your app.** Concretely:
- Your status-management UI should allow staff to move orders into
  `delivered` (from `paid`) or `failed` (from `pending`, e.g. for a
  no-show/abandoned order) — but should **never** offer a manual
  `pending → paid` action, and never let staff directly edit the `status`
  column to `'paid'` through any form field, bulk action, or API you build.
- If you need to confirm a payment yourself (e.g. reconciling a
  Paystack transaction that didn't get confirmed), do it the same way
  hungry-nomad does: call Paystack's verify endpoint server-side and only
  write `status = 'paid'` if the verified amount matches `total_amount`.
  Don't add a "mark as paid" button that skips verification.
- Reasonable transitions to build: `pending → failed` (manual, staff-
  initiated, no money involved), `paid → delivered` (manual, staff-
  initiated, no money involved). Anything that sets `status = 'paid'`
  should require actual verified payment, not a staff click.

## 3. Pricing/totals: read-only, never trust or recompute in your UI

`orders.total_amount` is computed once, server-side, at order-creation time
in hungry-nomad, from:
1. `products.price` for each line item, looked up fresh from the DB (never
   from client input),
2. `delivery_zones.fee` for the order's `delivery_lga`,
3. a flat `TAKEAWAY_FEE` (300 Naira) added if any item's `products.category`
   is `'regular'` or `'chinese'` (fast food, ice cream, and beverages don't
   trigger it).

`order_items.price_at_time` is a permanent snapshot — it does not update if
`products.price` changes later.

For your app: **treat `total_amount` and `price_at_time` as immutable,
authoritative historical facts.** Don't add an "edit total" field, don't
recompute a total from current `products.price` and display that instead,
and don't let staff override either value. If you need to show "what would
this cost today," compute that as a clearly-labeled separate figure, not by
mutating the order.

## 4. What you can and can't call in hungry-nomad

hungry-nomad exposes exactly two order-related endpoints, both customer-
facing, both **not for you to call**:
- `POST /api/orders` — order creation (used by the customer checkout flow).
- `GET /api/orders/[id]?reference=...` — order lookup gated by a
  Paystack-reference match, used by hungry-nomad's own post-payment success
  page. It deliberately returns only a subset of columns and won't work
  without the reference.

**There is no status-update endpoint, no bulk-order-list endpoint, and no
staff auth of any kind left in hungry-nomad.** Do not build your app to call
into hungry-nomad's API for admin operations — it has nothing to offer you
there. Instead, talk to Supabase directly, using your own credentials (§1),
for everything staff-side: listing orders, viewing details, updating
`status`, etc.

## 5. Auth for your staff users

hungry-nomad has no staff auth anymore — you're free to design your own.
One approach that existed in hungry-nomad before removal, worth considering
as prior art (not a mandate):

- Use Supabase Auth for staff login (email/password or magic link).
- Tag staff accounts with `app_metadata.role = 'staff'` (set via the
  Supabase Admin API or dashboard — `app_metadata` isn't user-editable, so
  it's safe to trust in RLS policies, unlike `user_metadata`).
- Write RLS policies on `orders`/`order_items` that check
  `auth.jwt() -> 'app_metadata' ->> 'role' = 'staff'` for the operations
  your staff needs (`SELECT`, `UPDATE` on `status`, etc.).
- This lets you skip a service-role key entirely and rely on RLS + real
  logged-in staff sessions instead — arguably safer, since a leaked
  service-role key bypasses RLS completely, while a leaked staff JWT is at
  least scoped to whatever policies you defined.

If you'd rather use a service-role key for simplicity (server-side only,
e.g. Next.js API routes / server actions never exposed to the browser),
that's also fine — just provision your own key, don't ask for hungry-
nomad's, and never let it anywhere near client-side code.

## 6. Environment variables you'll need

Your own `.env`, separate from hungry-nomad's:

```
NEXT_PUBLIC_SUPABASE_URL=            # same Supabase project URL as hungry-nomad
NEXT_PUBLIC_SUPABASE_ANON_KEY=       # same anon key is fine to reuse for anything anon-readable (products, delivery_zones) — useless for orders/order_items given default-deny RLS
SUPABASE_SERVICE_ROLE_KEY=           # YOUR OWN — do not ask for or reuse hungry-nomad's. Server-only, never NEXT_PUBLIC_, never in client components.
```

Provision separately, not shared with hungry-nomad:
- Your own Sentry project/DSN if you want error tracking (don't mix into
  hungry-nomad's project).
- Your own Resend key/from-address if you send staff notification emails.
- Your own rate-limiting store (e.g. Upstash Redis) if you need it — don't
  reuse hungry-nomad's Redis instance or key prefixes
  (`ratelimit:orders:create`, `ratelimit:orders:track` are its namespace).

Things you almost certainly don't need: `PAYSTACK_SECRET_KEY` (you're not
initiating or normally re-verifying payments — read `status`/
`payment_reference` instead, unless you're specifically building a
reconciliation feature per §2), `NEXT_PUBLIC_PAYSTACK_PUBLIC_KEY` (no reason
a staff dashboard opens the Paystack widget).

## 7. Quick-reference summary

- Same Supabase project, tables `products` / `delivery_zones` / `orders` /
  `order_items`. Column names/types are in §1.
- `orders`/`order_items` are RLS default-deny for `anon` — bring your own
  service-role key or your own staff-role Supabase Auth setup.
- Status values: `pending | paid | failed | delivered`. Only hungry-nomad
  (via real Paystack verification) should ever set `paid`. Your app owns
  `failed` and `delivered`. **Never build a manual "mark as paid" action.**
- `total_amount` and `price_at_time` are server-computed and historical —
  read-only in your UI, never editable by staff.
- hungry-nomad has no API for you to call for admin work. Go straight to
  Supabase with your own credentials.
- Staff auth is entirely your design — Supabase Auth + `app_metadata.role =
  'staff'` + RLS is a reasonable, previously-used pattern, not a
  requirement.
